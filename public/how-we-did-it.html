How We Found Out Massachusetts Spent $152 Million Digitizing Its Deed Records

The complete story of a question that started in ChatGPT, moved through Claude, and ended as a published interactive website — in about five hours.

March 1, 2026
Participants: Tom Scheffey • Nate Scheffey (Prince Street Consulting) • ChatGPT (OpenAI) • Claude & Claude Code (Anthropic)

Chapter 1: The Question

The project began when Nate's father, Tom Scheffey, was having a conversation with ChatGPT about a specific question: how much has Massachusetts spent digitizing its Registry of Deeds records? Since 2002, the state has been scanning and indexing physical deed books to make them searchable online through a system called MassLandRecords. The effort is funded by a $5 surcharge on every document recorded at any of the state's 21 registries. But nobody had ever added up the total cost.

ChatGPT identified a promising data source — the Massachusetts Open Checkbook portal (CTHRU) — and determined that the relevant spending was tracked under a fund called the "County Registers Technological Fund" (Fund Code 0050). ChatGPT wrote a detailed research plan and even produced a Python script to pull the data. But the script required a developer's environment to run, and Tom isn't a programmer. So he brought Nate in.

Chapter 2: Nate Reframes the Project for Claude

Rather than simply running ChatGPT's script, Nate recognized that this was really a multi-phase research and data analysis project — something better suited to an AI agent that could access the internet, write and run code, iterate when things broke, and produce polished deliverables. He decided to route the project through Claude.

Nate's first step was to take the ChatGPT-generated research plan and submit it to Claude (via the claude.ai chat interface) for evaluation:

Nate
See the below prompt. Tell me if this project would be best in Claude chatbot (here) or in Claude code, or as a fully git repo'd GSD Claude code project?

Below that instruction, he pasted the full research prompt — a detailed 12-section document that ChatGPT had produced, covering the objective (compute the total statewide digitization cost from 2002 to present), the data source (the CTHRU API), scope definitions, a classification methodology with keyword filters and vendor clustering, analytics requirements, quality controls, and seven specific deliverable files.

Claude
Claude recommended Claude Code (a command-line AI agent that can write and execute code, access the internet, and produce files) as the best environment. The reasoning: the chatbot couldn't do the API calls and file generation needed; a full Git-managed project was overkill for a one-shot research task. Claude Code was the sweet spot — it could handle the full pipeline in a single session with the ability to adapt on the fly. Claude also suggested testing the API endpoint first, before committing a full session to it.

Chapter 3: Nate Has Claude Test the API

Before handing the prompt to Claude Code, Nate wanted to verify that the data source actually worked:

Nate
Yes test it

Claude ran a battery of web searches and attempted to fetch data from the API. This testing phase uncovered five significant problems with ChatGPT's original prompt:

1. Wrong domain. The prompt pointed to cthrurevenue.mass.gov (the revenue portal). The correct endpoint for spending data was cthruspending.mass.gov.

2. An official typo. The fund is actually named "County Registers Techological Fund" in the state system — missing the 'n' in Technological. Legislative filings use the correct spelling, but the fund code itself carries the misspelling.

3. Uncertain API parameters. The prompt assumed specific filter parameters (org3, Fund_Code) that couldn't be verified as working. Claude Code would need to experiment or fall back to downloading unfiltered data.

4. Data only goes back to FY2010. The CTHRU portal explicitly states that expenditure data starts at Fiscal Year 2010, not 2002. Eight years of data were missing from the proposed approach.

5. Wrong API parameter. The pay_types=pay1 parameter was from the payroll portal, not the spending portal.

Without this testing step, Claude Code would have wasted significant time discovering these issues during execution.

Chapter 4: Nate Commissions the Corrected Prompt

With the five issues identified, Nate asked Claude to produce a corrected version of the analysis prompt — one that incorporated all the fixes and could be handed directly to Claude Code:

Nate
Yes update the prompt to include these issues explicitly. Then give it to me as an Md file I can hand to Claude code.

Claude produced a revised 13-section markdown document (ma-deeds-digitization-prompt.md) that corrected the API domain, removed the bad parameter, added confirmed URL patterns from testing, documented the FY2010 data floor with supplementary source strategies for earlier years, flagged the parameter mapping uncertainty with a fallback approach, noted the official typo, and included the actual CSV column schema. It also added a list of supplementary data sources to investigate for the missing FY2002–2009 period: legislative filings, Secretary of State reports, MassOpenBooks, and direct Comptroller records.

Chapter 5: Nate Launches the Analysis in Claude Code

Nate saved the corrected prompt file to his Desktop, opened Claude Code (a terminal-based AI agent), and gave it a single instruction:

Nate
read the ma-deeds-digitization-prompt.md and execute all instructions. ask me if anything seems unclear or suboptimal

Claude Code read the prompt and came back with four practical questions: where to save the output files (it suggested ~/Desktop/ma-deeds-analysis/), whether to prioritize local filtering over API parameter guessing, how much effort to invest in the FY2002–2009 data gap, and whether "manual review" steps should be done programmatically. Nate's response was concise:

Nate
proceed with your recommendations

From this point, Claude Code ran autonomously for approximately 25 minutes.

Phase A: Data Acquisition (~10 minutes)

Claude Code began by testing the CTHRU spending API with multiple parameter combinations, confirming what the earlier testing had predicted: server-side fund filtering did not work. Parameters like org3, org4, and Fund_Code were all either ignored or returned unfiltered data. One test returned a 45MB file — the entire state spending dataset for a single year.

Claude Code then discovered that filtering by department (org2=SECRETARY+OF+STATE+(SEC)) worked and returned a manageable 2.8MB file per year. It also made a critical correction: the relevant fund code was 0049, not 0050 as ChatGPT's original prompt had stated. The fund name in the actual data was "REGISTERS TECHNOLOGICAL FUND" (correctly spelled, unlike the legislative records).

Using this approach, Claude Code downloaded Secretary of State spending data for every fiscal year from FY2010 through FY2026, then filtered each file locally for Fund Code 0049. The result: 11,103 transactions totaling $61.8 million.

Phase B: Classification (~5 minutes)

The prompt's keyword-based classification approach (searching for terms like "scanning," "imaging," "backfile") turned out to capture almost nothing — only $84,000 out of $61.8 million. The reason: Massachusetts' state accounting system (MMARS) uses generic object codes like "U05 - Information Technology Professionals" and "U07 - IT Equipment Purchase" rather than descriptive text about what was purchased.

Claude Code adapted by building an object-code-based classification system instead. It mapped each state MMARS object code into three tiers: Narrow (codes directly tied to scanning and imaging, like J33 — Micrographic Services), Broad (narrow plus all IT infrastructure like equipment, software, and contracted tech staff), and Excluded (payroll, legal, office supplies, facilities). It validated the classification by sampling 100 records.

The results: $3.27 million narrow (direct scanning/imaging), $37.65 million broad (all IT infrastructure), and $24.17 million excluded (payroll and operations).

Phase C: The Socrata Discovery

While the classification was running, Claude Code had launched a parallel background agent to search for the missing FY2002–2009 data. This agent searched legislative filings, the Comptroller's website, and open data portals. It made a breakthrough discovery: a Socrata open data API hosted at cthru.data.socrata.com contained a "Non-Budgeted Special Revenue Funds" dataset with fund-level annual totals going all the way back to FY2005.

Even more significant: the Socrata data revealed that there were actually two parallel funds, not one. Fund 0049 ("Registers Technological Fund") covered the 15 state-controlled registry districts, while Fund 0050 ("County Registers Techological Fund" — with the misspelling) covered the 6 county-controlled districts (Barnstable, Bristol, Dukes, Norfolk, Plymouth, and Nantucket). The CTHRU transaction-level data only captured Fund 0049. Fund 0050 had been entirely invisible until this discovery.

This roughly doubled the scope of the findings. Claude Code pulled the complete Socrata dataset and recalculated: $151.9 million in combined spending across both funds from FY2005 to FY2026 ($207.1 million adjusted for inflation to 2026 dollars).

Phase D: Legislative PDFs

The background research agent also tracked down five legislative PDF filings — spending plans that individual registries had submitted to the state legislature under M.G.L. Chapter 29, Section 2KKK(a). The most valuable was SD2501, a statewide spending plan from FY2016 that contained per-registry breakdowns. This was the only source of district-level spending detail, since the state accounting system lumps all transactions under "Secretary of State" with no registry-level coding.

Claude Code downloaded all five PDFs, extracted their content using a PDF parsing library (the files were scanned images requiring special handling), and incorporated the findings into the analysis.

Phase E: Report Generation

Claude Code generated all seven deliverable files: a master dataset of all 11,103 transactions, a classified version with narrow/broad flags, annual totals with inflation adjustments, a vendor analysis ranking 125 vendors, a full analysis report, an executive summary, and a technical appendix documenting the complete methodology.

When Claude Code reported completion, Nate reviewed the output and made one additional request:

Nate
save the socrata data to the project directory too. also pull down the PDFs you used, including the SD2501. I want all referenced data alongside the reports.

Claude Code saved the raw Socrata API response, the five legislative PDFs, and all 17 yearly CTHRU CSV files into a source-data/ directory alongside the reports. The complete analysis package was now on Nate's Desktop.

Chapter 6: Nate Commissions the Website

With the analysis complete, Nate wanted to make the findings accessible to the public. He went back to Claude (the chatbot) and uploaded the entire analysis output as a zip file. He had already drafted a rough website prompt and wanted Claude to refine it before handing it to Claude Code:

Nate
claude code did a great job and now i have a report and all source data downloaded (attached). i now want to build a website to convey this data to interested readers. ive written the following prompt in claude code but i want you to optimize it before i submit:

Nate's draft prompt read:

Nate's draft
take the data in the ma-deeds-analysis folder and use it to build a website conveying the project, goals and findings, how it was prompted and accomplished, and anything else that may be of interest to a reader. the web pages should be in a clean format, similar to a newspaper article. the article should provide citations and links to the source data, and allow the user to view that without leaving the site. set up a github repo and deploy via vercel. include visual aids if helpful, e.g. mermaid charts, data visualization. ask me any questions before you start that will help make this project more successful. the ultimate goal of the project is to inform the reader. after initial questions proceed through with complete website creation and deploy.

Claude read every file in the uploaded analysis package — the executive summary, full report, technical appendix, all CSVs, source data structure, and legislative PDFs — and produced a significantly more detailed version. The optimized prompt specified exact section-by-section content with data mappings, defined the tech stack (Next.js static export, Tailwind CSS, Recharts for visualization), separated files into render-inline versus download-only based on actual file sizes, referenced ProPublica and The Marshall Project as design models, specified serif body text with maximum content width and muted chart palettes, instructed Claude Code to bake data into the build as static JSON rather than loading CSVs client-side, and included five specific pre-flight questions.

Chapter 7: Claude Code Builds and Deploys the Site

Nate pasted the optimized prompt into a new Claude Code session. Claude Code read all the source data and asked its five pre-flight questions:

Claude Code
1. Domain: Do you want a custom domain or is the default .vercel.app URL fine for now?

2. Emphasis: Is there anything in the data you want emphasized or de-emphasized? For example, should the vendor concentration finding or the two-fund structure get more prominence?

3. Additional context: Do you want any additional framing added — comparison to other states, editorial perspective, or broader commentary on government IT spending?

4. Attribution: Are you the credited author, and if so, how should you be attributed?

5. Other preferences: Any content, branding, or structural preferences? Color palette, logo, specific fonts, social sharing metadata?

Nate answered them all in a single message:

Nate
1. default fine. 2. no emphasis decisions, just present the information most informatively and clearly. 3. not at this point. 4. dont credit it at this point. 5. no

Claude Code then entered a planning phase, designed the full site architecture (directory structure, component hierarchy, data processing pipeline), and after Nate approved the plan, built the entire site: a data processing script that converted CSVs into typed TypeScript data files, five chart components using Recharts, sortable data tables, expandable markdown sections for the full report and methodology, download links for all source data files, and a responsive single-page layout styled as a long-form investigative data article.

The build hit a few technical snags along the way — a Recharts compatibility issue with Next.js server components, type annotation errors in chart formatters, and a Vercel deployment failure when the build script tried to access source data files that weren't present on the server — but Claude Code resolved each one autonomously without needing Nate's input.

The site went live at ma-deeds-digitization.vercel.app
